Complete Order Types, TimeInForce & Statuses

📋 Comprehensive DSE/CSE Market Coverage

Every order type the DSE and CSE markets support — including those that must be rejected with proper error codes. The FIX tag references follow FIXT.1.1 / FIX 5.0 SP2 protocol.

Order Categories (OrdType — FIX Tag 40)

Limit
Tag 40=2 · DSE ✅ · CSE ✅
Execute only at specified price or better. The most common order type. Stays in order book until filled, cancelled, or expired.
Market
Tag 40=1 · DSE ✅ · CSE ✅
Execute immediately at the best available price. No price guarantee. Used when speed is priority. Requires opposite orders to exist.
Market at Best
Tag 40=Z · DSE ✅ · CSE ✅
Execute at the best available opposite price. If no opposite order exists, rejected (FIX cert M1). If partially filled, remaining converts to limit at execution price (FIX cert M5).
Stop Loss
Tag 40=3 · DSE ⚠️ · CSE ⚠️
Triggers a market order when stop price is reached. Supported in system, may be exchange-limited. Store stop price in StopPrice field.
Stop Limit
Tag 40=4 · DSE ⚠️ · CSE ⚠️
Triggers a limit order when stop price is reached. Two price fields: StopPrice (trigger) + LimitPrice (execution ceiling). More controlled than Stop Loss.
GTC / GTD
Tag 59=1/6 · DSE ❌ · CSE ❌
Good Till Cancel / Good Till Date. Must be REJECTED by our system with DSE error message per FIX cert L18: "GTC/GTD orders are not supported." Stored as rejected with proper error code.

Time In Force (FIX Tag 59) — Controls Order Lifetime

Day (DAY)
Tag 59=0 · Default · DSE ✅ · CSE ✅
Valid for the current trading session only. Auto-cancelled at session close. Default TimeInForce if not specified.
IOC — Immediate or Cancel
Tag 59=3 · DSE ✅ · CSE ✅
Fill as much as possible immediately, cancel the rest. Partially fills are accepted. Unfilled quantity expires. FIX cert M6, L5, L6, L11.
FOK — Fill or Kill
Tag 59=4 · DSE ✅ · CSE ✅
Fill the entire quantity immediately or cancel the whole order. No partial fills accepted. FIX cert M9, M10, L4, L12. DSE reports TimeInForce=FOK even when sent as IOC with MinQty=OrderQty (L3).
AON — All or None
Tag 18=G (ExecInst) · DSE ⚠️ · CSE ⚠️
Fill entire quantity or nothing. Unlike FOK, order can rest in book waiting for full fill. Implemented via ExecInst=G (All or None). Required for Block board via ExecInst=WholeOrNone (FIX cert B7).
Session (AT_THE_OPEN)
Tag 59=2 · Pre-Open ✅
Orders placed during Pre-Open session. Status becomes Queued. At market open, Queued → Submitted → matching. FIX cert L19.
GTC — Good Till Cancel
Tag 59=1 · DSE ❌ REJECTED
Not supported by DSE. System must reject with proper FIX error. Stored as Rejected with OrdRejReason code.
GTD — Good Till Date
Tag 59=6 · DSE ❌ REJECTED
Not supported by DSE. System must reject with proper FIX error. Stored as Rejected with OrdRejReason code and rejection reason text.

Special Order Instructions (ExecInst — FIX Tag 18)

Private / Suspend
Tag 18=S · DSE ✅ (optional feature)
Order placed as private — not released to market. Sits as Private (Status=Z) until trader releases it. FIX cert M2, M4, L13, B8, B9.
Release
Tag 18=q · DSE ✅ (OrderCancelReplaceRequest)
Amend a private order to release it to market. Sent as OrderCancelReplaceRequest with ExecInst=q. OrderID changes after release. FIX cert M4 step 4.
Iceberg / Drip
Tag 1138=DisplayQty · DSE ✅ · CSE ✅
Show only DisplayQty to market. When filled, refill from hidden quantity. Rejected on Block board (FIX cert B6). FIX cert L14, L17.
Whole or None (Block)
Tag 18=M (WholeOrNone) · Block Board only
Required on Block board. Without this, order is rejected (FIX cert B7). Equivalent to AON for block trades.
Minimum Quantity
Tag 110=MinQty · DSE ✅ · CSE ✅
Minimum acceptable fill. Must be equal to OrderQty for IOC (otherwise rejected, FIX cert L2). FIX cert L3, M7, M8.

Target Architecture

🔗 FIX Connector Strategy

Interface: IFIXConnector with methods for all message types.
Default: SimulatedFIXConnector — faithfully implements all FIXT.1.1 messages, all cert test scenarios.
Live: QuickFIXNConnector — configure SenderCompID, TargetCompID, host, port in Admin Settings → FIX Config. Zero code changes to order flow. Just swap via FIXConnectorFactory reading config.

🏢 Multi-Tenant Strategy

Now: Single DB BdStockOMS. Row-level isolation via BrokerageHouseId on every table. Every service reads tenantId from ITenantContext from JWT only.
Future-ready: BrokerageConnections model + ITenantDbContextFactory interface from Day 80. SuperAdmin UI can provision per-tenant DB. Architecture in place — enable per-brokerage DB by config.

⚡ Live Data Strategy

Layer 1: REST API on initial widget mount (full data).
Layer 2: SignalR subscription for real-time updates (delta).
Layer 3 (fallback): Last-fetched data from widget state if SignalR drops — shown with "stale" badge.
Layer 4 (fallback): Redis-cached data served by API if scraper fails.
Never empty. Never broken-looking.

🌍 i18n Strategy (from Day 100)

Library: react-i18next.
Files: public/locales/en/translation.json, bn/translation.json, zh/translation.json.
Toggle: Language selector in Topbar, persists to localStorage.
Coverage: All 108 existing files patched Day 100. New components use t('key') from Day 100 onward.
Format: Intl API for date/number formatting per locale.

Design Principles — Apply to Every Line

🔗 Everything Connected

  • Every widget: REST load → SignalR updates
  • Last-fetched fallback if API fails (not demo data — real last data)
  • Market closed: show last session data with "Closed" indicator
  • Loading skeleton on initial fetch only
  • Retry button only if truly no data available

⚡ Optimistic UI

  • Update UI immediately on action
  • Confirm from server async
  • Revert with error toast if server rejects
  • p95 < 50ms market data API
  • p95 < 200ms order placement
  • Redis cache first, DB fallback

🔐 Multi-Tenant Default

  • BrokerageHouseId from JWT on every query
  • Never from request body
  • Per-tenant feature flags
  • SuperAdmin can query any tenant
  • Audit log includes tenantId always

🎛️ Micro-Level Control

  • RMS limits: per-stock/sector/board/category/user/BO-group
  • Column show/hide/resize/reorder — persisted per user per widget
  • Excel download on every table
  • Right-click context menu on every table row
  • Every form: keyboard shortcuts

🧪 Test Everything

  • Every new service: xUnit tests same day
  • 0 failures always — non-negotiable
  • Vitest: 40+ component tests (from Day 109)
  • Playwright: 10 E2E journeys (Day 110–111)
  • NBomber: 200 concurrent users (Day 112)
  • Target: 1,600+ tests at go-live

📋 Documented Always

  • Every service: XML doc comments
  • Every endpoint: Swagger annotations
  • Every FIX tag: referenced in code comment
  • Every trading rule: documented in code + docs
  • Every keyboard shortcut: registered in SHORTCUTS constant

Non-Negotiable Development Rules

🔒 These Rules Never Change
  • OrderType = Buy | Sell (the SIDE). OrderCategory = Market | Limit | MarketAtBest | StopLoss | StopLimit (the TYPE). Never confuse.
  • OrderStatus.Filled — not "Executed". The word Executed is banned from the codebase.
  • Single DB: BdStockOMS. Row-level isolation via BrokerageHouseId from JWT on every query. Never from request body.
  • SignalR Hub: StockPriceHub. Signal: TradeExecuted.
  • Auth store: @/store/authStore. Not useAuthStore, not AuthContext.
  • DTO namespace: BdStockOMS.API.Models.Admin
  • AppDbContext path: BdStockOMS.API/Data/AppDbContext.cs — NOT Models/
  • Frontend files: cat > file << 'ENDOFFILE' heredocs in Git Bash.
  • C# files with !: use python -c list-of-strings approach.
  • Run set +H before bash with ! characters.
  • Stop API before build if locked: taskkill //F //IM BdStockOMS.API.exe
  • One branch per day. dotnet test must pass before commit. 0 failures always.
  • Live data on every widget. Last-fetched fallback if API fails. Never empty.
  • English only until Day 100. After Day 100: all strings via t('key').
  • Every keyboard shortcut registered in the central SHORTCUTS constant (for Feature Master List auto-generation).
  • Every trading rule documented in code comment AND in relevant SOP document.
  • Every FIX tag referenced by tag number in code comment: // Tag 40=2 (Limit)
